# 기술명세서 (설계 폴더)

## 1. 문서 목적
- 본 문서는 `Blazor_Embedded_v2` 저장소의 현재 솔루션 구조를 기준으로 시스템 구현 방향, 기술 표준, 품질 기준을 설계 관점에서 정리한다.
- 특히 `BlazorApp_arduinoSearch_240824_01`의 장비 탐색, MQTT 연동, UI 제어 흐름을 중심으로 기술한다.

## 2. 문서 범위
- 아키텍처 계층(프레젠테이션, 서비스, 데이터 모델, 정적 리소스)
- 통신 흐름(HTTP 장비 정보 조회, MQTT 발행/구독, JS 인터롭)
- 운영 제약(Electron 모드, 로컬 네트워크 탐색, 포트/주소 정책)
- 품질 요구(안정성, 복구성, 성능, 보안, 유지보수성)
- 테스트/빌드 운영 기준(솔루션 빌드 가능 상태, 경고 현황 관리)

## 3. 솔루션 인벤토리
- `BlazorApp_arduinoSearch_240824_01`: 스마트팜 장비 탐색 및 MQTT 제어 중심 서버사이드 Blazor 앱
- `BlazorApp3`: 기본 Blazor Server 템플릿 기반 실험 솔루션
- `BlazorChart240928`: 차트 시각화 중심 Blazor 앱 샘플
- `MudBlazorWebApp240916`: MudBlazor 기반 구성 실험 솔루션
- `mudBlazorApp_v01`: Client/Server/Shared 분리형 MudBlazor 솔루션

## 4. 핵심 기술 스택
- ASP.NET Core / Blazor Server (.NET 6 대상 포함)
- Razor Components / Layout / Routing
- MQTTnet 기반 MQTT 클라이언트
- HttpClient 기반 로컬 네트워크 장비 API 호출
- Chart.js + JavaScript Interop
- MatBlazor UI 컴포넌트
- Electron.NET 하이브리드 실행

## 5. 아키텍처 설계 원칙
- UI와 통신 로직의 역할 분리(페이지/컴포넌트 ↔ 서비스)
- 장비 탐색/연결/제어 흐름을 사용자 시나리오 단위로 명확화
- 네트워크 실패를 정상 시나리오로 간주하고 오류 가시성 확보
- 토픽 구조를 동적으로 수용해 디바이스 확장성 확보
- 실시간 데이터 반영 시 UI 재렌더링 비용 최소화
- 브라우저 렌더링 자산(JS 차트) 수명주기 정합성 유지
- 배포 환경(Electron/웹) 차이에 대한 구성 일관성 확보

## 6. 컴포넌트 책임 정의
- `Program.cs`: DI 등록, 파이프라인 구성, Electron 부트스트랩, 바인딩 주소 지정
- `DeviceDiscoveryService`: IP 범위 탐색, Ping 검사, 장비 메타데이터 수집
- `MqttService`: MQTT 연결/구독/발행, HTTP 구성 전달, 메시지 이벤트 브로커 역할
- `Pages/Devices.razor`: 장비 목록 UI, 사용자 입력 수집, 연결 트리거, 모달 제어
- `Pages/LedControlComponent.razor`: LED 제어 + 조도 데이터 시각화 컨트롤
- `wwwroot/js/lightSensorChart.js`: 조도 누적 막대차트 렌더링 및 업데이트 처리
- `DataModel/Device.cs`: 장비/토픽/트리 노드 모델 정의
