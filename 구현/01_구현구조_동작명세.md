# 구현 구조 및 동작 명세 (구현 폴더)

## 1. 목적
- 현재 저장소의 구현 파일 기준으로 실행 구조, 런타임 흐름, 책임 분리를 정리한다.

## 2. 구현 루트 구조
- `BlazorApp_arduinoSearch_240824_01`
- `BlazorApp3`
- `BlazorChart240928`
- `MudBlazorWebApp240916`
- `mudBlazorApp_v01`

## 3. 핵심 실행 흐름 개요
- 애플리케이션 시작 시 Program.cs에서 서비스 등록 및 미들웨어 구성 수행
- Devices 페이지 진입 시 DeviceDiscoveryService가 네트워크 대역을 스캔
- 사용자 선택 장비에 대해 MqttService가 HTTP 구성 전달 및 MQTT 연결 수행
- 메시지 수신 시 컴포넌트 이벤트로 UI 상태 및 Chart.js 데이터 갱신
- 모달 종료/페이지 전환 시 자원 정리와 상태 복원 수행

## 4. 파일별 구현 책임
- `Program.cs`: 현재 동작에 직접 영향을 주는 구현 파일
- `App.razor`: 현재 동작에 직접 영향을 주는 구현 파일
- `Pages/Devices.razor`: 현재 동작에 직접 영향을 주는 구현 파일
- `Pages/LedControlComponent.razor`: 현재 동작에 직접 영향을 주는 구현 파일
- `Pages/LedControl.razor`: 현재 동작에 직접 영향을 주는 구현 파일
- `Services/MqttService.cs`: 현재 동작에 직접 영향을 주는 구현 파일
- `DeviceDiscoveryService.cs`: 현재 동작에 직접 영향을 주는 구현 파일
- `DataModel/Device.cs`: 현재 동작에 직접 영향을 주는 구현 파일
- `wwwroot/js/lightSensorChart.js`: 현재 동작에 직접 영향을 주는 구현 파일
- `wwwroot/js/sensorChart.js`: 현재 동작에 직접 영향을 주는 구현 파일
- `Pages/_Host.cshtml`: 현재 동작에 직접 영향을 주는 구현 파일

## IM-001 구현 시나리오
- 시작 조건: 사용자가 구현 시나리오 001를 트리거한다.
- 입력 데이터: 디바이스 정보, MQTT 서버/포트, 토픽 맵, 사용자 액션.
- 처리 단계 1: UI 레이어에서 유효성 검사 및 서비스 호출 준비.
- 처리 단계 2: 서비스 레이어에서 네트워크 통신/예외 처리 수행.
- 처리 단계 3: 수신 데이터 모델 변환 및 상태 저장.
- 처리 단계 4: 컴포넌트 상태 반영 및 렌더링 갱신.
- 처리 단계 5: 로그 및 오류 메시지 기록.
- 종료 상태: 성공/실패/재시도 가능 상태로 수렴.
- 구현 포인트: 비동기 컨텍스트 교착 회피, null 참조 방지, 타입 일치.
- 검증 포인트: 버튼 상태, 모달 상태, 차트 데이터 반영, 연결 상태 일관성.
